# Internationalization

Last updated: 2026-06-09

NestWeb officially supports two backend locales:

- `zh`
- `en`

The old `fr` email locale has been removed. Backend email verification now uses Chinese for `country=CN` and English for other countries.

## Scope

Backend i18n currently covers:

- email templates under `src/i18n/*/email.json`
- standard API envelope messages through `TransformInterceptor`
- common HTTP / Prisma error messages through `HttpFilter`
- externalized business errors that have keys under `src/i18n/*/common.json`

Language resolution supports, in order, `?lang=`, `x-custom-lang`, and
`Accept-Language`. Unsupported or missing languages fall back to `zh`.

Not every future business string is automatically bilingual. New modules must
add matching `zh` and `en` keys first, then throw or return key-backed messages
instead of hard-coded UI-facing strings.

## Rules

- Keep only `src/i18n/zh` and `src/i18n/en`.
- Add matching JSON files to both locales.
- Keep nested translation keys aligned between Chinese and English.
- Do not add a third locale unless product support is explicitly required.
- Use `zh` as the backend fallback language, matching the frontend default.
- Do not return raw Prisma or infrastructure errors to clients.
- Prefer key-backed errors for user-facing business exceptions.

## Checks

Run:

```bash
pnpm run i18n:check
```

The check enforces:

- only `zh` and `en` exist under `src/i18n`
- both locales contain the same JSON files
- both locales contain the same nested translation keys

## Adding A New Locale Later

1. Copy `src/i18n/en` to `src/i18n/<locale>`.
2. Translate every value.
3. Update `scripts/check-i18n.cjs` `allowedLocales`.
4. Update locale selection rules in services that choose a language explicitly.
5. Run `pnpm run i18n:check`, `pnpm run lint:check`, `pnpm test -- --runInBand`, and `pnpm run build`.
